# Diff UI spec

## Metadata

- **Feature name**: `awp diff UI`
- **Owner**: AI coding agent
- **Status**: Done
- **Last updated**: 2026-04-10

## Goal
Give users a polished, daily-driver diff viewer for the current Jujutsu repo so they can review changed files and hunks, refresh quickly, search, and jump directly into their editor at the relevant changed line.

## User Problem
The user wants the "nice diff viewer" experience from an earlier prototype, but adapted to the current awp redesign. Today awp has workspace lifecycle commands and some Bubble Tea UI pieces, but no dedicated review UI. Reviewing changes currently requires bouncing between raw `jj diff`, ad hoc editor commands, and manual navigation. That is slower and less pleasant than a purpose-built TUI.

## Scope
### In scope (v1)
- Add a new top-level command: `awp ui`.
- Run the diff viewer against the **current repo / current working copy** only.
- Launch directly in the current terminal session (not via tmux-managed panes/windows).
- Show a changed-files list and hunk preview.
- Support keyboard navigation for files and hunks.
- Support opening the selected file/hunk in `$EDITOR` (or editor fallback) at the relevant changed line.
- Support filtering/searching the changed file list.
- Support manual refresh.
- Support manual refresh for a polished daily-driver experience.
- Handle common diff cases well: modified files, new files, deleted files, renames, and empty diff state.
- Add/update automated tests for parser, UI model behavior, editor command generation, and CLI wiring.

### Out of scope (v1)
- Tmux-managed embedding or automatic pane/window creation.
- Multi-workspace/session browsing.
- Interactive patch editing/staging/reverting from the TUI.
- Side-by-side diff rendering.
- Git support outside jj repos.
- Persisted UI preferences/history.
- Inline comments, PR review actions, or code-ownership metadata.

## UX
### CLI
- New command: `awp ui`
- Running `awp ui` outside a jj repo returns an actionable error.
- Running `awp ui` inside a jj repo opens a full-screen Bubble Tea TUI.
- No workspace argument is supported in v1; the command operates on the current repo context.

### TUI
- Header shows that the user is in the awp diff UI and indicates the current repo/workspace context where practical.
- Left panel lists changed files.
- Right panel shows the selected file's hunks.
- Footer/help shows keybindings and refresh status.
- Required interactions:
  - `j/k` and arrow keys move through files/hunks
  - `tab` switches focus between files and hunks; `h`/`l` and left/right also switch focus between panels
  - `enter` or `e` opens the selected file or hunk in the editor at the changed line
  - `/` enters filter mode for file search
  - `r` refreshes the diff immediately
  - `r` refreshes the diff immediately
  - `q` quits
- Empty state clearly indicates when there are no changes.
- Errors are visible but non-destructive when possible (for example, refresh failure should not crash the program).

## Discovery Questions
1. Who is the first user? Andrew, working inside jj repos and wanting a polished review workflow in awp.
2. When do they use this feature? During normal coding/review loops, after making changes or while inspecting in-progress work.
3. What exact output/result do they need? A full-screen diff browser with fast keyboard navigation and one-keystroke jump into the editor.
4. What data sources are required? `jj diff --git` for detailed hunks, current repo detection from `jj root`, and environment/editor configuration from `$EDITOR`/fallback behavior.
5. What is the smallest useful slice? `awp ui` for the current repo with file list, hunk preview, editor jump, filter, and refresh.
6. What are explicit non-goals? Tmux integration, multi-workspace scope, and patch-manipulation features.
7. What does “done” look like? A stable top-level `awp ui` command that feels polished enough for daily use in the current repo.

- 2026-04-10: Refined keyboard ergonomics (`h`/`l`, `e`) and stabilized the filter UI with a reserved footer row so opening the filter does not shift the overall layout or corrupt surrounding rendering.

## Implementation Plan
1. Add jj adapter support for obtaining git-format diff output for the current repo context, with explicit, testable command behavior.
2. Add a diff parser package/model that converts `jj diff --git` output into file/hunk structures with changed-line mapping for editor jumps.
3. Add an editor adapter that resolves `$EDITOR`/fallback behavior and constructs commands for opening files at a given line.
4. Add a Bubble Tea diff UI package with:
   - file-list focus
   - hunk-list focus
   - filter mode
   - manual refresh
   - auto-refresh
   - clear status/error rendering
5. Wire `awp ui` into the existing CLI app as a top-level command.
6. Add tests for CLI command behavior, jj command generation, diff parsing edge cases, editor command generation, and UI model interactions.
7. Polish copy, empty states, and error messages so the feature is suitable for daily use.
8. Keep refresh behavior predictable; prefer manual refresh over disruptive background updates.
